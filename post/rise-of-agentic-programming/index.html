The Rise of Agentic Programming: A New Era of Software Development

25 Apr 2025, 00:00
technical / software / development / AI / agents / MCP

The evolution of AI-assisted programming has reached a pivotal moment, transforming from simple code completion to truly collaborative AI agents. This shift represents not just an improvement in tooling, but a fundamental change in how software is developed. Or is it?

The Early Days: From Completion to Comprehension

The journey began with GitHub Copilot in June 2021, which introduced AI-powered code completion that could suggest entire functions. While revolutionary at the time, it was fundamentally reactive - waiting for developers to write code before making suggestions. In February 2023, the Cursor editor launched with the first implementation of true “agent mode” capabilities, demonstrating that AI could maintain context across entire conversations about code. This was followed by Amazon CodeWhisperer’s integration of conversation memory in late 2023, and Anthropic’s Claude extension for VS Code in early 2024, which brought sophisticated project-wide understanding to mainstream IDEs.

What made these early agents different was their ability to maintain context across entire conversations and understand project-wide implications of changes. However, they still struggled with complex architectural decisions and often needed extensive prompting to produce reliable code.

The 2024 Breakthrough

The major technical breakthrough in mid-2024 came from advances in LLMs. Multiple folks directly building LLMs are saying that there is a step increase in capability EVERY TWO MONTHS. But what happened in 2024 that really changed things? Claude 3.5 things it’s three intersecting advances:

1. Neural-Symbolic Integration:

* Models gained the ability to actually simulate code execution paths
* They could understand how code would behave in different contexts
* They could detect potential runtime issues before execution
* Integration with build systems allowed real-world validation

2. Advanced Context Processing:

* New attention mechanisms specifically designed for code structure
* Ability to maintain context across multiple services and repositories
* Understanding of complex dependency chains
* Project-wide awareness of architectural patterns

3. Tool Integration Architecture:

* Deep integration with build systems and CI/CD pipelines
* Real-time analysis of deployment specifications
* Understanding of package management and dependencies
* Integration with security scanning tools

This wasn't just an incremental improvement - it represented a fundamental shift in how AI models understood and interacted with code. The models moved from pattern matching against training data to actually reasoning about code behavior and system architecture.

These advances laid the foundation for the Model Context Protocol (MCP) and Agent-to-Agent (A2A) specifications

I haven’t fact checked that, but it seems not far off. Basically, 2024 was the year that LLMs started to be able to really do code.

The Protocol Revolution: MCP and A2A

The adoption of the Model Context Protocol (MCP) enables AI agents to actually DO THINGS. Like go gather data. And write files. Or hit an API to make something happen. And effectively build a workflow.

The Agent-to-Agent (A2A) specification (just released last month) takes this further, enabling multiple specialized agents to collaborate on complex tasks. It claims it can enable:

  • One agent might focus on architectural decisions
  • Another could specialize in security reviews
  • A third might handle test generation
  • While others could focus on documentation and maintenance

This is all foundational stuff. I do wonder if MCP will be the new SOAP (credit to Scott Francis for that insight) but time will tell. We’ll adapt, just like we did when REST came out.

What Tools?

Today’s development environment is radically different from just last winter. Major platforms have embraced agentic programming:

  • Cursor Editor
  • Visual Studio Code GitHub Copilot integration
  • JetBrains
  • WundSurfer (formerly Codium)

All of these can be basically used to write entire applications nearly autonomously. But… will they work? Maybe.

My Early Results

I’ve build a few smaller projects, in C, go and typescript. None of them were of significant size. The time savings to at least get a working scaffolding is insane. It’s at least 100x.

Do I completely trust it? No. Of course not. On my team we still require a second code review and that cannot be done by AI.

Does it write the code the way I would? Rarely. In fact, the code I’ve seen it write does not really follow the design patterns that I would use. But why do we use design patterns? We use them for two reasons:

  • they are tried and true patterns to solve a problem
  • in a way that other humans can grok when looking at the code later

But… for a lot of modules, there’s no real need for a human to grok the code. If you really embrace this and have the AI write your tests first (you do write tests first, right?) and if you review those tests carefully and if the AI-generated code is passing the tests… then do you really care it didn’t use the design pattern you would have?

Because now you can:

inspect the code in this repository and write a design document about how it works.  Your audience is an experienced software engineer.  Generate block diagrams, sequence diagrams and state diagrams as needed using mermaid.

Try that. You’ll be amazed.

What’s Next for Me?

Keep testing this! Embrace! Learn what it can and cannot do. Folks, it’s real. It’s not going away. And I’ll tell you about it as I go.
